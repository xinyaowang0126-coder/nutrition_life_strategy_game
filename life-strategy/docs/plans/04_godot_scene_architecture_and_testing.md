# Godot 场景架构与测试工作方案

## 目标

让项目继续建立在 Godot 的场景、资源、主题、信号和编辑器工作流上，而不是由一个脚本动态生成所有界面。目标是让功能可独立调试、界面可在编辑器中调整、数据与结算可测试。

## 本轮已落地

- 使用 Godot 编辑器/MCP 创建并保存了 `ChoiceCard.tscn`、`PhaseChoiceView.tscn` 和 `ToastFeed.tscn`，节点结构可在编辑器中直接查看和调整。
- 创建 `GameTheme.tres`，将基础字号、颜色与按钮样式从运行时代码移入主题资源。
- `ToastFeed` 已作为真实子场景实例接入 `GameRoot.tscn`。
- 抽牌与用餐结算分别拆入 `MealDeckService` 和 `MealResolver`，减少界面脚本直接处理规则。
- 卡牌、来源和库存数据继续由项目数据文件加载，来源字段与时段字段已经分离。
- 自动化测试已覆盖数据完整性、固定随机、去重、库存过滤、费用、组合和关键结局。
- 本轮测试已通过 13 项用例，并使用运行中的 Godot 项目检查了来源选择界面。

## 仍需后续

### P0：稳定当前拆分

- 为三个组件场景补充独立预览或最小测试场景，编辑 UI 时不必每次进入完整七天流程。
- 清除组件初始化对入树时机的隐含依赖，所有公开方法都应在节点就绪前后表现一致或明确限制。
- 把来源选择、食物选择、行动选择、睡眠选择的状态切换写成明确状态机，禁止散落的字符串分支继续增长。
- 统一通过信号传递“选中、确认、返回、跳过”，由 `GameRoot` 只负责流程编排。

### P1：继续缩小 `GameRoot`

- 拆出 `RunState`：持有天数、金钱、指标、库存和每日记录。
- 拆出 `PhaseController`：负责阶段顺序、可返回范围与一天结束。
- 拆出 `HudView` 和 `EndingView` 场景：资源显示、每日摘要与结局展示可独立编辑。
- 将背景切换封装为 `SceneBackdrop`，以资源表映射时段和地点。
- 将来源、食物、行动、睡眠逐步迁移为自定义 `Resource` 或经过校验的数据对象；XML 保留为内容源时，应有单独导入和验证层。
- 减少运行时设置尺寸、边距和颜色的代码，把稳定的布局属性留在 `.tscn` 与 Theme 中。

### P1：扩展测试层次

- 单元测试：所有结算函数、组合阈值、结局优先级、库存和费用边界。
- 场景测试：组件实例化、信号、禁用状态、焦点移动和重复配置。
- 流程测试：从新游戏走完一天和七天，覆盖返回、跳过、钱不足、存粮耗尽、重开。
- 数据验证：每张卡牌必须有有效图片、来源、时段、数值范围和唯一 ID。
- 回归记录：固定至少三组随机种子，保存关键状态快照以发现平衡逻辑的意外变化。

### P2：建立编辑器友好的内容流程

- 为常用数据增加 Godot Inspector 可编辑资源或导入工具，并在保存时立即报告错误。
- 为卡牌提供预览场景，内容编辑者能直接查看标题、图片、短摘要与悬停详情。
- 将多分辨率截图、测试运行和数据校验整理成发布前检查脚本。
- 在导出 Web/桌面版本前自动运行测试，并检查缺失资源与解析警告。

## 迭代阶段

1. 稳定阶段：现有场景无初始化错误，流程状态清晰，测试保持全绿。
2. 解耦阶段：`GameRoot` 只做编排，状态、规则、视图分别负责自己的职责。
3. 工具阶段：内容和 UI 可在编辑器内预览、校验，发布检查可重复执行。

## 验收标准

- 主流程 UI 的稳定结构可以在 Godot 场景树中编辑，不依赖运行时从零创建。
- `GameRoot` 不直接实现抽牌、餐费、库存、组合或结局算法。
- 每个主要组件有清晰输入、输出信号和独立验证方式。
- 改变 Theme 或组件场景即可影响所有同类界面，不需要搜索多处分散样式代码。
- 自动化测试覆盖核心边界，固定种子流程可重复；测试、运行日志均无新错误。
- Web 与桌面导出前完成一轮完整流程和规定分辨率的视觉验收。
